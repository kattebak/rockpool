## Summary

Introduce a `rockpool` CLI (`npx rockpool <command>`) that becomes the single entry point for running and configuring the rockpool stack. The CLI consolidates the three configuration layers (config file, compose env vars, npm scripts) into one: the config file. Ports, runtime, auth, and all other settings live in `rockpool.config.json`. The CLI generates a compose file on the fly and invokes `podman compose`.

## Motivation

The current setup has three places a user must understand to configure rockpool:

1. **Config file** (`rockpool.config.json`) — auth, runtime, log level, SPA serving
2. **Compose env vars** — ports (`SRV0_PORT`, `SRV1_PORT`, `SRV2_PORT`), queue port, DB path
3. **npm scripts** in `package.json` — glue that sets the right env vars per profile

This means port configuration lives in env vars and npm scripts, while auth lives in the config file. The user must cross-reference multiple files to understand the full picture.

A CLI solves this by:

- **One config file.** Ports move into the config schema alongside auth and runtime. Everything is in one place.
- **One command.** `npx rockpool run` replaces `npm start` / `npm stop` / env var juggling.
- **Guided setup.** `npx rockpool init` creates a valid config file interactively, lowering the barrier for new users.
- **Generated compose.** The compose file is derived from the config at invocation time. No more maintaining env var mappings by hand.

## CLI Commands

### `npx rockpool run [config-file]`

Start the rockpool stack.

1. Load and validate config file (default: `rockpool.config.json` in cwd)
2. Build the client if needed (see [Client Build](#client-build) below)
3. Generate `compose.yaml` in a temp location (or `.rockpool/compose.yaml`)
4. Run `podman compose -f <generated> up`
5. On `SIGINT`/`SIGTERM`, run `podman compose -f <generated> down`

```bash
npx rockpool run                         # uses rockpool.config.json
npx rockpool run rockpool.test.config.json  # uses test config
```

### `npx rockpool stop [config-file]`

Stop the running stack. Derives the compose project name from the config file path so it stops the correct stack.

### `npx rockpool init`

Interactive config file creation. Supports two modes:

**Interactive (default)** — walks the user through each config section:

```
? Authentication mode (basic / github): basic
? Username: admin
? Password: ****
? HTTP port [8080]: 8080
? IDE port [8081]: 8081
? Preview port [8082]: 8082
? Log level [info]: info
? Runtime (podman / stub) [podman]: podman
? SPA mode (proxy / static / none) [none]: proxy
? Vite dev server URL [http://localhost:5173]: http://localhost:5173

Created rockpool.config.json
```

**Non-interactive (flags)** — all options can be passed as CLI flags for automation:

```bash
npx rockpool init \
    --auth-mode basic \
    --auth-username admin \
    --auth-password admin \
    --port-http 8080 \
    --port-ide 8081 \
    --port-preview 8082 \
    --log-level info \
    --runtime podman \
    --spa-proxy-url http://localhost:5173
```

When all required fields are provided via flags, no prompts are shown. When some flags are provided, only missing fields are prompted for. This makes `init` usable in scripts and CI while remaining friendly for interactive use.

Validates each answer against the Zod schema. Writes the result as JSON with a `$schema` reference for IDE support.

### `npx rockpool logs [config-file]`

Tail compose logs for the running stack.

## Config Schema Changes

Add a `ports` section to the existing schema:

```typescript
const PortsSchema = z.object({
    http: z.number().int().min(1).max(65535).default(8080),
    ide: z.number().int().min(1).max(65535).default(8081),
    preview: z.number().int().min(1).max(65535).default(8082),
    caddy: z.number().int().min(1).max(65535).default(2019),
});

export const RockpoolConfigSchema = z.object({
    logLevel: LogLevelSchema.default("info"),
    runtime: RuntimeSchema.default("podman"),
    server: ServerSchema.default({}),
    auth: AuthSchema,
    spa: SpaSchema.default({}),
    urls: UrlsSchema.optional(),
    ports: PortsSchema.default({}),      // new
});
```

All ports have defaults matching the current behavior. Existing config files without `ports` continue to work unchanged.

## Client Build

The CLI must ensure the client is built before starting the stack. When `spa.proxyUrl` is set, the Vite dev server handles serving — no build needed. Otherwise, the stack serves static files from `packages/client/dist`, which must exist.

On `rockpool run`, the CLI checks whether a build is needed:

1. If `spa.proxyUrl` is set → skip build (Vite dev server will serve the client)
2. If `packages/client/dist` exists and is newer than `packages/client/src` → skip build (already up to date)
3. Otherwise → run `npm run build -w packages/client` before starting compose

```typescript
function needsClientBuild(config: RockpoolConfig, projectRoot: string): boolean {
    if (config.spa.proxyUrl) return false;

    const distDir = join(projectRoot, "packages/client/dist");
    const srcDir = join(projectRoot, "packages/client/src");

    if (!existsSync(distDir)) return true;

    const distMtime = statSync(distDir).mtimeMs;
    const srcMtime = statSync(srcDir).mtimeMs;

    return srcMtime > distMtime;
}
```

This means `npx rockpool run` works out of the box after `npm install` — no manual build step. The build runs once and is cached until source files change.

The `init` command hints at this:

```
? SPA mode (proxy / static) [static]: static

Note: the client will be built automatically on first run.
```

## Compose Generation

The CLI generates a compose file by injecting config values into a template. The template is a TypeScript object serialized to YAML — no string templating.

```typescript
import { stringify } from "yaml";

function generateCompose(config: RockpoolConfig, projectRoot: string): string {
    const { ports } = config;

    const compose = {
        services: {
            caddy: {
                image: "docker.io/library/caddy:2",
                command: "caddy run --config /etc/caddy/Caddyfile --adapter caddyfile",
                ports: [
                    `${ports.caddy}:${ports.caddy}`,
                    `${ports.http}:${ports.http}`,
                    `${ports.ide}:${ports.ide}`,
                    `${ports.preview}:${ports.preview}`,
                ],
                environment: {
                    CADDY_ADMIN_PORT: String(ports.caddy),
                },
                restart: "unless-stopped",
                volumes: [
                    "caddy-data:/data",
                    "caddy-config:/config",
                    `${projectRoot}/Caddyfile:/etc/caddy/Caddyfile:ro`,
                ],
            },
            elasticmq: {
                image: "docker.io/softwaremill/elasticmq-native",
                restart: "unless-stopped",
                volumes: [
                    `${projectRoot}/elasticmq.conf:/opt/elasticmq.conf:ro`,
                ],
            },
            "control-plane": {
                build: `${projectRoot}/images/control-plane`,
                init: true,
                working_dir: "/app",
                environment: {
                    ROCKPOOL_CONFIG: `/app/${configFileName}`,
                    CONTAINER_HOST: "unix:///run/podman.sock",
                    CONTROL_PLANE_HOST: "control-plane",
                    PORT: "7163",
                    CADDY_ADMIN_URL: `http://caddy:${ports.caddy}`,
                    SRV0_PORT: String(ports.http),
                    SRV1_PORT: String(ports.ide),
                    SRV2_PORT: String(ports.preview),
                    DB_PATH: "rockpool.db",
                    QUEUE_ENDPOINT: "http://elasticmq:9324",
                    QUEUE_URL: "http://elasticmq:9324/000000000000/workspace-jobs",
                    CONTAINER_HOST_ADDRESS: "host.containers.internal",
                    SPA_PROXY_URL: config.spa.proxyUrl || "",
                },
                security_opt: ["label=disable"],
                restart: "unless-stopped",
                volumes: [
                    `${projectRoot}:/app`,
                    "node-modules:/app/node_modules",
                    "rockpool-data:/opt/rockpool",
                    "/var/run/docker.sock:/run/podman.sock",
                ],
                depends_on: ["caddy", "elasticmq"],
            },
        },
        volumes: {
            "caddy-data": null,
            "caddy-config": null,
            "rockpool-data": null,
            "node-modules": { name: "rockpool-node-modules" },
        },
    };

    return stringify(compose);
}
```

Generation is a pure function: config in, YAML string out. No file I/O, no side effects, easy to test.

## Package Structure

New package: `packages/cli/`

```
packages/cli/
├── package.json
├── src/
│   ├── bin.ts              # entry point, arg parsing
│   ├── commands/
│   │   ├── run.ts          # load config, generate compose, exec podman
│   │   ├── stop.ts         # stop the running stack
│   │   ├── init.ts         # interactive config creation
│   │   └── logs.ts         # tail compose logs
│   ├── compose.ts          # generateCompose() pure function
│   └── prompt.ts           # interactive prompt helpers for init
└── test/
    ├── compose.test.ts     # snapshot tests for generated compose
    └── init.test.ts        # prompt flow tests
```

### package.json

```json
{
    "name": "@rockpool/cli",
    "version": "0.0.1",
    "type": "module",
    "bin": {
        "rockpool": "./src/bin.ts"
    },
    "dependencies": {
        "@rockpool/config": "workspace:*",
        "yaml": "^2"
    }
}
```

The `bin` entry points directly at the `.ts` file. The shebang uses Node 22's native TypeScript support:

```typescript
#!/usr/bin/env -S node --experimental-strip-types --no-warnings
```

No build step. `npx rockpool` works directly from the workspace.

### Argument Parsing

Use `node:util` `parseArgs` — no external dependency needed for the simple command structure:

```typescript
import { parseArgs } from "node:util";

const { positionals } = parseArgs({
    allowPositionals: true,
});

const [command, ...args] = positionals;
```

### Interactive Prompts (`init`)

Use Node 22's built-in `node:readline/promises` for interactive input. No external prompt library needed:

```typescript
import { createInterface } from "node:readline/promises";

const rl = createInterface({ input: process.stdin, output: process.stdout });
const mode = await rl.question("Authentication mode (basic / github) [basic]: ");
```

For password input, disable echo via raw mode on the TTY.

## Impact on Existing Code

### Config schema

Add `PortsSchema` to `packages/config/src/schema.ts`. Fully backward compatible — the `ports` field has defaults for all values.

### compose.yaml

The checked-in `compose.yaml` remains for reference and for users who prefer raw compose. The CLI generates its own compose file and does not modify the checked-in one.

### npm scripts

Update root `package.json` to use the CLI:

```json
{
    "start": "rockpool run",
    "stop": "rockpool stop",
    "logs": "rockpool logs",
    "test:e2e:headless": "rockpool run rockpool.test.config.json -- npx playwright test"
}
```

The existing npm scripts become thin wrappers around the CLI, preserving backward compatibility.

### E2E tests

The E2E global setup uses `rockpool init` to generate the test config, then `rockpool run` to start the stack. This eliminates the hand-maintained `rockpool.test.config.json` and exercises the CLI's init and run commands on every E2E run.

```typescript
// e2e/global-setup.ts
import { execSync } from "node:child_process";

const TEST_CONFIG = "rockpool.test.config.json";

export default async function globalSetup(): Promise<void> {
    execSync(`npx rockpool init \
        --auth-mode basic \
        --auth-username test \
        --auth-password test \
        --port-http 9080 \
        --port-ide 9081 \
        --port-preview 9082 \
        --runtime podman \
        -o ${TEST_CONFIG}`, { stdio: "inherit" });

    execSync(`npx rockpool run ${TEST_CONFIG}`, { stdio: "inherit" });

    await pollUntilReady("http://localhost:9080/api/health", 60_000);
}
```

```typescript
// e2e/global-teardown.ts
export default async function globalTeardown(): Promise<void> {
    execSync(`npx rockpool stop ${TEST_CONFIG}`, { stdio: "inherit" });
}
```

Benefits:

- **No config file to maintain.** The test config is generated, not checked in. If the schema changes (new required field, renamed option), the init invocation is the single place to update.
- **CLI is tested on every E2E run.** Both `init` (non-interactive) and `run`/`stop` are exercised. A broken CLI means broken E2E — fast feedback.
- **CI and local use the same path.** No separate env files or npm script glue per profile.

## Testing Strategy

### Unit Tests

- `compose.test.ts` — snapshot tests: given a config, assert the generated YAML matches expected output. Cover default ports, custom ports, with/without SPA proxy, with/without cloudflare tunnel.
- `init.test.ts` — test prompt validation logic (port range, auth mode validation) without actual TTY interaction. Test that flags bypass prompts and that partial flags prompt only for missing fields.

### Integration Tests

- E2E global setup exercises `init` (non-interactive) + `run` + `stop` on every test run (see [E2E tests](#e2e-tests) above).
- If E2E passes, the CLI works end-to-end.

### Manual Verification

```bash
# Interactive init
npx rockpool init
# → prompts for each field, creates rockpool.config.json

# Non-interactive init
npx rockpool init --auth-mode basic --auth-username admin --auth-password admin
# → creates rockpool.config.json with defaults for unprovided fields

# Run
npx rockpool run
# → builds client if needed, starts stack on configured ports

# Stop
npx rockpool stop
```

## Rollout Plan

### Phase 1: Config schema + compose generation

1. Add `PortsSchema` to `@rockpool/config`
2. Regenerate JSON schema (`npm run generate-schema -w packages/config`)
3. Create `packages/cli/` with `generateCompose()` function
4. Unit tests for compose generation

### Phase 2: CLI commands

1. Implement `run`, `stop`, `logs` commands
2. Wire up signal handling for graceful shutdown
3. Update npm scripts to use CLI
4. Verify E2E tests pass through CLI

### Phase 3: Interactive init

1. Implement `init` command with readline prompts
2. Schema-driven validation of user input
3. Write config file with `$schema` reference

## Open Questions

- [ ] **Compose file location.** Write to `.rockpool/compose.yaml` (persistent, inspectable) or a temp directory (no cleanup needed)? Leaning toward `.rockpool/` so users can inspect the generated file for debugging.
- [ ] **Project name derivation.** Compose needs a project name to isolate stacks. Derive from config file name (`rockpool-test` for `rockpool.test.config.json`) or from a field in the config?
- [ ] **Cloudflare tunnel.** The current compose has a `cloudflared` service behind a `tunnel` profile. Should the CLI support `--tunnel` flag, or should this be a config field?
- [ ] **Watch mode.** The current compose passes `NODE_WATCH` env var for `--watch` support. Should the CLI always enable watch in dev, or make it a config/flag?
